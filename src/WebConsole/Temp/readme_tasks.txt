	1. Диалоговое окно для настроек сложных фильтров (TaskOptionsDialog). Валидация и табы. 
У BasicDialog extjs'a есть свойство autotabs. Если поставить его в true, тогда div'ы вида
	<div class="x-dlg-tab" title="Title">
            <!-- Nested "inner-tab" to safely add padding -->
            <div class="inner-tab">
                 Info
            </div>
        </div>
будут автоматически отображаться как табы. Надо уделить внимание разметке\css, потому что выглядит коряво.
	При валидации с VallidationCalloutExtender возникает следующая проблема: из-за корявового кода как в extjs, так и в AjaxControlToolkit всплывающее окно об ошибке рисуется для элемента в скрытом табе притом в неправильной позиции. 
У меня времени уже не было менять, но там есть два варианта: либо override'ить методы TabPanel для extjs, либо менять код VallidationCalloutExtender2 в AjaxControlToolkitX (библиотека, которую добавляли чтобы отображать сплывающее окно в разных позициях). 
	В первом варианте нужно копать в волшебной разметке ext-1.1.1\source\widgets\TabPanel.js строка 524. Там при скрывании таба выполняется
	hideAction : function(){
        	this.bodyEl.hide();
        	this.bodyEl.setStyle("position", "absolute");
       	 	this.bodyEl.setLeft("-20000px");
        	this.bodyEl.setTop("-20000px");
    	}.
Зачем-то скрытый таб сдвигают непонятно куда, пробовал быстро менять - не получилось. Вероятно тащит с собой ещё что-то. Надо будет изучить полностью этот файл, чтоб разобраться.
	Второй вариант, видимо более простой, в яваскрипте VallidationCalloutExtender2 добавить код, который не будет показывать всплывающее окно если элемент или один из его родителей по цепочке скрыт (visibility: hidden). 

	Вторая проблема связана с необходимостью переключаться на таб, в котором произошла ошибка валидации. В ссылке описанорешение для AjaxControlToolkit табов:
http://sandblogaspnet.blogspot.com/2009/04/setting-focus-to-particular-tab-in.html
Для  extjs будет похоже, надо будет только нужное Api вызывать.

Если на странице присутствуют несколько CustomizableTask контролов с соответствующими TaskOptions иногда возникают проблемы с валидацией (Появляется в неправильном месте сообщение об инвалидности текстового поля из другого TaskOptions). Вероятно потому, что TaskOptions не прячутся, как должны при выставление Visible=false при асинхронном обновлении. Нашёл слишком поздно, чтобы фиксить. Можно поменять имя группы валидации и передавать его TaskOptionsDialog (немного коряво но просто). Либо копать почему так происходит.

Как добавлять новые сложные задания:
В проекте Tasks пишется класс расширяющий TaskEntity используя необходимые артибуты по примеру CreateProcessTaskEntity. Потом по примеру контрола CreateProcessTaskOptions пишется свой контрол с опциями.
На TaskPanel на конечной странице (Default3) кидается ещё один CustomizableTask (смотреть по аналогии). Туда же кидается свой контрол с опциями, на который указывает добавленный CustomizableTask.  На TaskPanel обязательно должны присутствовать TaskOptionsDialog.
